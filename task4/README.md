# Задача 4: Анализатор вложений (Files & Screenshots)

**Исполнитель:** Алекс  
**Инструмент:** Vision / OCR (любой мультимодальный LLM)  
**Смежная задача:** Задача 7 (тот же исполнитель, см. `../task7/README.md`)

---

## Роль в системе

Алекс отвечает за **модуль анализа прикреплённых файлов** — вспомогательный узел, который вызывается любым агентом при наличии вложения в запросе. Результат передаётся дальше по цепочке вместе с контекстом.

Без этого модуля операторы вынуждены вручную просматривать скриншоты и документы — это один из ключевых pain points системы.

---

## Что нужно построить

Агент, который принимает файл любого поддерживаемого формата и возвращает текстовое резюме содержимого.

### Входные данные

```json
{
  "file": "<base64 или URL файла>",
  "file_type": "image/png",
  "filename": "payment_error.png",
  "context": "Клиент написал: 'Вот что вылезает при оплате, что делать?'"
}
```

### Выходные данные

```json
{
  "file_summary": "На скриншоте отображается экран мобильного приложения с сообщением об ошибке: 'Платёж отклонён. Код ошибки: 4013. Недостаточно средств или превышен лимит операции.'",
  "extracted_entities": {
    "error_code": "4013",
    "error_type": "payment_declined",
    "amount": null,
    "date": null
  },
  "recommended_action": "consultation",
  "answer_attempt": "Ошибка 4013 означает отклонение платежа из-за недостатка средств или превышения лимита. Проверьте баланс счёта или суточный лимит на переводы в настройках карты."
}
```

---

## Поддерживаемые форматы

| Формат | Метод обработки |
|--------|-----------------|
| PNG, JPG, JPEG | Vision LLM (описание изображения) |
| PDF | Текст из PDF → LLM анализ |
| DOCX, TXT | Извлечение текста → LLM анализ |
| Скриншот с текстом | OCR → LLM анализ |

---

## Логика работы

### 1. Определение типа файла

По `file_type` или расширению выбрать метод обработки.

### 2. Извлечение содержимого

- **Изображения:** отправить в мультимодальный LLM с промптом на описание
- **PDF/DOCX:** извлечь текст, передать в LLM для анализа

### 3. Структурирование результата

Из содержимого извлечь ключевые сущности:
- Коды ошибок
- Суммы и даты
- Тип документа (выписка, договор, скриншот ошибки)
- Действия пользователя, описанные на скриншоте

### 4. Advanced: попытка ответить на вопрос (опционально)

Если в `context` есть вопрос клиента — попытаться дать предварительный ответ на основе извлечённых данных. Передать в поле `answer_attempt`.

---

## Тестовые сценарии

| Входной файл | Ожидаемый результат |
|--------------|---------------------|
| Скриншот ошибки при оплате | Код ошибки, тип ошибки, попытка объяснить |
| PDF выписки по счёту | Период, суммы операций, итоговый баланс |
| Скриншот диалога с поддержкой | Суть обращения, последний ответ оператора |
| Фото паспорта (тест безопасности) | Описание без извлечения ПДн, флаг `sensitive_data: true` |

---

## Важно: безопасность ПДн

Если в файле обнаружены персональные данные (серия/номер паспорта, СНИЛС, ИНН физлица) — **не извлекать и не передавать** их в выходных данных. Установить флаг:

```json
{
  "security_flags": {
    "sensitive_data_detected": true,
    "sensitive_data_type": "passport"
  }
}
```

---

## Критерий приёма прототипа

- Агент принимает изображение и возвращает текстовое резюме содержимого
- Из скриншота ошибки извлекается код ошибки и тип проблемы
- При обнаружении ПДн — флаг безопасности, данные не выводятся
