# Задача 7: Аналитик для менеджмента (Summary Agent)

**Исполнитель:** Алекс  
**Инструмент:** Evo Agents  
**Смежная задача:** Задача 4 (тот же исполнитель, см. `../task4/README.md`)

---

## Роль в системе

Алекс также отвечает за **аналитического агента** — финальный узел цепочки. Агент не взаимодействует с клиентами напрямую: он обрабатывает накопленные логи жалоб и формирует структурированный отчёт для менеджмента.

Этот модуль решает один из ключевых pain points — отсутствие системного анализа причин жалоб. Без него менеджмент не видит тенденций и не может принимать взвешенные решения.

---

## Что нужно построить

Агент пост-обработки, который принимает массив завершённых тикетов и генерирует управленческий отчёт.

### Входные данные

```json
{
  "period": {
    "from": "2026-04-01",
    "to": "2026-04-07"
  },
  "complaints": [
    {
      "id": "ticket_001",
      "category": "payment_delay",
      "client_type": "B2C",
      "resolved": true,
      "justified": true,
      "compensation": 1500,
      "satisfaction_score": 4,
      "summary": "Задержка платежа на 2 дня"
    }
  ]
}
```

### Выходные данные

```
Отчет по жалобам
Период: 01.04–07.04.2026

Резюме:
За неделю обработано: 142 жалобы, из них 68% обоснованы.
Основная причина: задержки в проведении платежей (31%).

Ключевые метрики:
- Всего жалоб: 142
- Обоснованных: 97 (68%)
- Выдано компенсаций: 83 на сумму 245 000 руб
- Средняя оценка после решения: 4.2/5

ТОП-3 причины:
1. Задержки платежей: 31%
2. Ошибки в приложении: 24%
3. Сложности с верификацией: 18%

Рекомендации:
- Усилить мониторинг платежного шлюза
- Протестировать флоу верификации на мобильных устройствах
```

---

## Логика работы

### 1. Агрегация данных

| Метрика | Как считать |
|---------|-------------|
| Всего жалоб | `count(complaints)` |
| % обоснованных | `count(justified=true) / total * 100` |
| Выдано компенсаций | `sum(compensation) where compensation > 0` |
| Средняя оценка | `avg(satisfaction_score)` |

### 2. Группировка по категориям (топ-5)

```
payment_delay → «Задержки платежей»
app_error     → «Ошибки в приложении»
verification  → «Сложности с верификацией»
access_issue  → «Проблемы с доступом»
other         → «Прочее»
```

### 3. Генерация рекомендаций

На основе топ-категорий LLM формирует 2–3 конкретных рекомендации для команды.

Примеры правил:
- `payment_delay > 25%` → рекомендация проверить платёжный шлюз
- `app_error > 20%` → рекомендация провести тестирование мобильного приложения
- `verification > 15%` → рекомендация упростить флоу верификации

### 4. Форматирование отчёта

Отчёт генерируется в виде читаемого текста для менеджмента (не JSON). Стиль — деловой, конкретный, без технического жаргона.
