# Функциональная документация — Task 2 (B2C консультант, RAG Public)

## 1) Назначение

Модуль `process_b2c_consultation(payload)` обрабатывает консультационные запросы физлиц (B2C) и формирует безопасный ответ на основе публичной базы знаний.

## 2) Контракт функции

### Вход

```json
{
  "question_text": "Текст вопроса клиента",
  "dialog_context": {
    "messages_history": [
      {"role": "client", "text": "Предыдущее сообщение"},
      {"role": "agent", "text": "Предыдущий ответ"}
    ]
  },
  "client_type": "B2C",
  "topic": "consultation"
}
```

### Выход

```json
{
  "answer_text": "Готовый текст ответа",
  "answer_found": true
}
```

## 3) Логика обработки

1. **Проверка применимости сценария**
   - Если `client_type != B2C` или `topic != consultation`, возвращается `answer_found=false` и текст `NOT_APPLICABLE_TEXT`.
2. **Проверка пустого вопроса**
   - Пустой/пробельный запрос ведёт к fallback-ответу (`FALLBACK_TEXT`).
3. **Фильтр персональных/индивидуальных кейсов**
   - Если вопрос требует доступа к персональным данным (договор, списание, претензия и т.п.), возвращается `answer_found=false` и эскалация специалисту (`PERSONAL_DATA_FALLBACK_TEXT`).
4. **RAG Public (прототип)**
   - Извлекается текущий вопрос + история диалога.
   - Выполняется скоринг публичных чанков базы знаний с приоритетом текущего вопроса над историей.
   - При низком скоре возвращается fallback.
5. **Формирование ответа**
   - Для лучшего совпадения возвращается дружелюбный и простой текст, без внутренних терминов и без выдуманных условий.

## 4) Публичные источники знаний (эмуляция)

В прототипе база знаний зашита в коде и включает 4 группы:
- Тарифы и стоимость.
- Подключение и оформление.
- Личный кабинет и приложение.
- Ограничения и публичные условия.

## 5) Ограничения безопасности

Модуль **не**:
- возвращает данные по индивидуальным договорам;
- обрабатывает запросы на изменение персональных данных;
- выдаёт ответ, если в публичной базе недостаточно релевантной информации.

## 6) Набор тестов

См. `task2/test_b2c_consultant.py`.
Покрыты сценарии:
- корректная активация и деактивация;
- успешный ответ по тарифам;
- использование контекста для короткого follow-up вопроса;
- fallback для персональных кейсов;
- fallback для пустого вопроса;
- fallback для нерелевантного вопроса вне публичной базы.
