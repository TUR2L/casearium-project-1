# B2C консультант (RAG Public / EVO Managed RAG)

## 1. Назначение функции

**B2C консультант (RAG Public / EVO Managed RAG)** — функция агента для обработки типовых консультационных запросов от физических лиц на основе публичной базы знаний.

Функция предназначена для автоматической подготовки дружелюбного, понятного и корректного ответа клиенту с использованием публичных источников знаний: FAQ, документации по продуктам для физических лиц и правил тарификации.

Если агент не находит достаточно надёжный ответ в базе знаний, он не должен придумывать информацию. В этом случае функция возвращает флаг `answer_found: false` и формирует ответ с передачей обращения человеку-оператору.

---

## 2. Задача

Обработка типовых запросов физических лиц в рамках консультаций по продуктам, услугам, тарифам, условиям обслуживания и другим вопросам, информация по которым содержится в публичной базе знаний.

---

## 3. Условия активации

Функция активируется, если одновременно выполнены условия:

1. Клиент относится к сегменту **B2C** — физическое лицо.
2. Тема обращения определена как **консультация**.
3. Запрос не требует обязательного участия сотрудника на первом шаге.
4. Запрос может быть обработан на основании публичной базы знаний.

## 4. Источники знаний

Для прототипа используются публичные источники знаний:

1. **Публичный FAQ**  
   Частые вопросы и ответы для физических лиц.

2. **Документация по продуктам для физических лиц**  
   Описание продуктов, условий подключения, ограничений и сценариев использования.

3. **Правила тарификации**  
   Публичные правила расчёта стоимости, комиссии, тарифные условия и ограничения.

## 5. Входные данные

Функция принимает следующие данные:

```json
{
  "question_text": "Текст вопроса клиента",
  "dialog_context": {
    "messages_history": [
      {
        "role": "client",
        "text": "Предыдущее сообщение клиента"
      },
      {
        "role": "agent",
        "text": "Предыдущий ответ агента"
      }
    ]
  },
  "client_type": "B2C",
  "topic": "consultation"
}
```

### Описание полей

| Поле | Тип | Обязательность | Описание |
|---|---:|---:|---|
| `question_text` | string | Да | Текущий текст вопроса клиента. |
| `dialog_context` | object | Нет | Контекст диалога, если доступна история сообщений. |
| `messages_history` | array | Нет | История сообщений клиента и агента. Используется для уточнения смысла текущего вопроса. |
| `client_type` | string | Да | Тип клиента. Для активации функции значение должно быть `B2C`. |
| `topic` | string | Да | Тема обращения. Для активации функции значение должно быть `consultation`. |

## 6. Логика работы функции

### Шаг 1. Проверка условий активации

Агент проверяет, что обращение относится к сегменту B2C и теме консультации.

Если условия не выполнены, функция не используется, а обращение передаётся в другой подходящий сценарий обработки.

---

### Шаг 2. Подготовка поискового запроса

Агент анализирует:

- текст текущего вопроса клиента;
- контекст предыдущих сообщений, если он передан;
- ключевые сущности: продукт, услуга, тариф, действие, проблема, ограничение.

На основе этого формируется поисковый запрос для семантического поиска по публичной базе знаний.

---

### Шаг 3. Семантический поиск по базе знаний

Агент выполняет семантический поиск по подключённым публичным источникам:

- FAQ;
- документации по продуктам для физических лиц;
- правилам тарификации.

Цель поиска — найти фрагменты знаний, которые напрямую отвечают на вопрос клиента.

---

### Шаг 4. Оценка найденного ответа

Агент оценивает найденные материалы по критериям:

1. Ответ напрямую соответствует вопросу клиента.
2. Информация относится к физическим лицам.
3. Ответ основан на публичных источниках знаний.
4. Нет противоречий между найденными источниками.
5. Информации достаточно, чтобы дать клиенту полезный и безопасный ответ.

Если критерии выполнены, агент устанавливает:

```json
"answer_found": true
```

Если точного ответа нет или есть сомнения в корректности, агент устанавливает:

```json
"answer_found": false
```

---

### Шаг 5. Формирование ответа клиенту

Если ответ найден, агент формирует готовый текст ответа:

- в дружелюбном и спокойном тоне;
- простым языком, без сложных внутренних терминов;
- с кратким объяснением сути;
- при необходимости — с понятными шагами для клиента;
- без ссылок на внутренние системы и закрытые процессы;
- без выдуманных условий, тарифов или обещаний.

---

### Шаг 6. Передача человеку, если ответ не найден

Если ответ не найден, агент не должен придумывать информацию.

В этом случае агент формирует вежливый ответ клиенту и сообщает, что вопрос будет передан специалисту.

Пример ответа:

> Спасибо за вопрос. Я не нашёл точной информации в публичной базе знаний, поэтому передам обращение специалисту. Он проверит детали и поможет вам с ответом.

---

## 7. Выходные данные

Функция возвращает готовый ответ для клиента и флаг, показывающий, был ли найден ответ в базе знаний.

```json
{
  "answer_text": "Готовый текст ответа для клиента",
  "answer_found": true
}
```

### Описание полей

| Поле | Тип | Обязательность | Описание |
|---|---:|---:|---|
| `answer_text` | string | Да | Готовый текст ответа, который можно отправить клиенту. |
| `answer_found` | boolean | Да | `true`, если ответ найден в базе знаний; `false`, если требуется передача человеку. |

---

## 8. Требования к тону ответа

Ответ должен быть:

- дружелюбным;
- понятным для клиента без технической подготовки;
- спокойным и уверенным;
- кратким, но достаточным;
- без канцелярита;
- без внутренних терминов, если они не нужны клиенту.

### Пример хорошего тона

> Да, эту услугу можно подключить физическому лицу. Обычно для подключения нужно выбрать подходящий тариф и заполнить заявку. Стоимость зависит от выбранного тарифа и условий обслуживания. Если хотите, я могу подсказать, где посмотреть правила тарификации.

### Пример нежелательного тона

> Согласно регламенту обработки клиентских интентов B2C необходимо произвести валидацию тарифной матрицы и сверку с документацией.

---

## 9. Ограничения

Агент не должен:

1. Придумывать ответ, если информация не найдена.
2. Использовать закрытые или внутренние данные.
3. Давать юридически значимые обещания от имени компании.
4. Называть точную стоимость, если она не подтверждена в публичной базе знаний.
5. Отвечать на вопросы, требующие идентификации клиента или доступа к персональным данным.
6. Подменять человека в спорных, конфликтных или нестандартных ситуациях.

---

## 10. Правила fallback-сценария

Fallback-сценарий используется, если:

- ответ не найден в базе знаний;
- найденная информация противоречива;
- вопрос клиента неполный или неоднозначный;
- вопрос требует участия специалиста;
- вопрос связан с индивидуальным договором, персональными данными или претензией.

В fallback-сценарии агент возвращает:

```json
{
  "answer_text": "Спасибо за вопрос. Я не нашёл точной информации в публичной базе знаний, поэтому передам обращение специалисту. Он проверит детали и поможет вам с ответом.",
  "answer_found": false
}
```

---

## 11. Критерии успешности прототипа

Прототип считается успешным, если:

1. Агент подключён к публичному источнику знаний.
2. Агент корректно активируется для обращений B2C с темой консультации.
3. Агент выполняет семантический поиск по FAQ, документации и правилам тарификации.
4. Агент формирует понятный и дружелюбный ответ клиенту.
5. Агент не выдумывает ответы при отсутствии информации.
6. Агент возвращает корректный флаг `answer_found`.
7. При отсутствии ответа агент передаёт обращение человеку.

---

## 12. Результат для прототипа

В результате должен быть реализован агент с подключённым публичным источником знаний, который:

- принимает вопрос клиента и контекст диалога;
- определяет применимость сценария B2C-консультации;
- выполняет RAG-поиск по публичной базе знаний;
- формирует корректный ответ для клиента;
- возвращает флаг `answer_found: true`, если ответ найден;
- возвращает флаг `answer_found: false`, если требуется передача человеку.
